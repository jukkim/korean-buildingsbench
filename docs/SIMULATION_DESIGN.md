# Korean_BB 시뮬레이션 설계서

> ⚠️ **HISTORICAL (2026-02-19)** — 작성 당시 갭 +8%p는 초기 PatchTST 수치. 현재 Korean-700 val_best 13.11±0.16%(SOTA -0.16%p)로 목표 달성 완료. 시뮬 설계 자체는 그대로 유효하나 수치 목표값은 구버전.

> 작성일: 2026-02-19
> 목표: BB SOTA 13.31% 대비 갭(현재 +8%p)을 축소하는 한국형 시뮬레이션

## 1. 설계 원칙

BB가 13.31%를 달성한 핵심 요인은 **보정(calibration)이 아니라 규모와 다양성**이다.

| 요인 | BB (900K) | 우리 현재 | 목표 |
|------|-----------|----------|------|
| 건물 수 | 900,000 | ~1,000 | 10,000+ |
| 스케줄 아키타입 | 350K+ (인구 기반) | 320 (ASHRAE 고정) | 2,000+ (확률적) |
| baseload 범위 | 0.70-0.95 | 0.31-0.64 | 0.50-0.95 |
| 기후 다양성 | 미국 32개 조합 | 10개 도시 | 10개 도시 (충분) |
| 건물 연식 | 10+ vintage | 1 baseline | 5 vintage |

**핵심 전략**: 건물 수를 늘리는 것보다 **스케줄 다양성 + 연식별 외피 차이**가 더 중요하다.

## 5. 스케줄 다양성 (핵심 개선 사항)

### 현재 문제
기존 320개 ASHRAE 고정 스케줄 → 모든 사무실이 8-18시, 모든 병원이 24시간.
BB는 인구 기반 350K+ 고유 아키타입으로 자연스러운 다양성 확보.

### 해결: 확률적 스케줄 생성

각 건물 유형별 **스케줄 클래스** 정의 후, 클래스 내 확률적 변동 적용.

#### 5.1 사무실 스케줄 (5 클래스)

| 클래스 | 운영시간 | 점유율 피크 | 한국 특성 |
|--------|---------|:---:|------|
| 정부/공공 | 09:00-18:00 | 0.85 | 정시 퇴근, 금요일 조기 |
| 일반기업 | 08:30-19:00 | 0.80 | 야근 문화 (20% 잔류 ~21시) |
| IT/스타트업 | 10:00-22:00 | 0.70 | 유연출퇴근, 늦은 퇴근 |
| 콜센터/교대 | 08:00-20:00 | 0.90 | 2교대, 높은 점유율 |
| 소규모사무실 | 09:00-18:00 | 0.60 | 낮은 밀도, 외근 잦음 |

#### 5.2 주거 스케줄 (4 클래스)

| 클래스 | 출근/등교 | 귀가 | 한국 특성 |
|--------|---------|------|------|
| 맞벌이 | 07:30 출발 | 19:00 귀가 | 낮 시간 비어있음 |
| 외벌이/주부 | 08:00 (1인) | 19:00 | 주부 상시 재실 |
| 1인가구 | 08:30 출발 | 20:00 귀가 | 야식 문화, 늦은 취침 |
| 고령가구 | 상시 재실 | - | 높은 난방 수요, 낮 외출 (09-11시) |

#### 5.3 기타 건물 스케줄

| 건물 | 클래스 수 | 변동 요소 |
|------|:---:|------|
| 학교 | 3 | 초등(~15:30), 중고(~17:00+방과후), 대학(~22:00+야간) |
| 판매 | 4 | 대형마트(10-23), 백화점(10-21), 편의점(24h), 근린상가(10-19) |
| 병원 | 3 | 종합병원(24h), 의원(09-18), 요양원(24h 저밀도) |
| 호텔 | 3 | 비즈니스(높은 주중), 관광(높은 주말), 리조트(계절적) |

#### 5.4 확률적 변동 파라미터

각 스케줄 클래스에서 건물별 고유 변동 적용:

```python
schedule_params = {
    'arrival_time':    Normal(μ, σ=0.5h),     # ±1시간
    'departure_time':  Normal(μ, σ=0.75h),    # ±1.5시간
    'lunch_start':     Normal(12.0, σ=0.25h), # ±30분
    'lunch_duration':  Normal(1.0, σ=0.15h),  # 45-75분
    'peak_occupancy':  Beta(α, β) × max_occ,  # 건물별 밀도 차이
    'weekend_ratio':   Uniform(0.0, 0.3),     # 주말 운영 비율
    'night_occupancy': Uniform(0.02, 0.10),   # 야간 잔류율
    'seasonal_factor': {                       # 계절 변동
        'summer': Normal(1.0, 0.1),
        'winter': Normal(0.9, 0.1),
    },
    'holiday_handling': ['close', 'reduce', 'normal'],  # 공휴일 대응
}
```

#### 5.5 목표 스케줄 수

| 건물유형 | 클래스 | × 변동 | = 아키타입 |
|---------|:---:|:---:|:---:|
| 아파트 | 4 | ×100 | 400 |
| 사무실 | 5 | ×100 | 500 |
| 학교 | 3 | ×50 | 150 |
| 판매 | 4 | ×50 | 200 |
| 병원 | 3 | ×50 | 150 |
| 호텔 | 3 | ×50 | 150 |
| **합계** | **22** | | **1,550+** |

BB의 350K에는 못 미치지만, **320 → 1,550+ (5배 증가)** 로 다양성 대폭 개선.

## 6. Baseload 보정

### 현재 문제
SIM baseload: 0.31-0.64 vs BB real data: 0.70-0.95

### 원인 분석
- 기존 시뮬: 장비 밀도 낮음, 대기전력 미반영
- 한국 실제: 냉장고, 대기전력, 엘리베이터, 공용조명, IT장비 등

### 해결 방법

```python
baseload_components = {
    'apartment': {
        'refrigerator':     50,   # W (24h 상시)
        'standby_power':    30,   # W (TV, 공유기, 충전기 등)
        'common_lighting':  15,   # W/세대 (복도, 주차장)
        'elevator':         20,   # W/세대 (고층 기준)
        'ventilation':      10,   # W (HRV, V4-V5)
    },
    'office': {
        'server_room':      500,  # W (소규모 서버)
        'standby_power':    200,  # W (PC, 모니터 대기)
        'security_system':  100,  # W (CCTV, 출입통제)
        'common_lighting':  300,  # W (비상등, 복도)
        'elevator':         200,  # W
        'ventilation':      150,  # W (환기 팬)
    },
}
# 목표 baseload ratio: 0.50-0.85 (건물 유형/연식별 차등)
```

## 7. 시뮬레이션 매트릭스

### 조합 계산

```
7 아키타입 × 5 vintage × 10 도시 × 2 기상(TMY+AMY) = 700 base cases

각 base case에 스케줄 변동:
  - 아파트: 700 × 55% × (100 변동 / base) ≈ 보수적으로 총 ~5,000건
  - 비주거: 700 × 45% × (50 변동 / base) ≈ ~3,000건

총 시뮬레이션 수: ~8,000-10,000건
```

### 출력 포맷 (BB 호환)

```
output/
├── building_{id}/
│   ├── timeseries.parquet     # 8760h × [total_electricity_kWh]
│   ├── metadata.json          # 건물 정보 (type, vintage, city, HVAC, ...)
│   └── schedules.json         # 사용된 스케줄 파라미터
├── transforms/
│   ├── boxcox_params.json     # Box-Cox lambda (전체 건물 fitting)
│   └── scaler_params.json     # StandardScaler (weather)
└── index/
    ├── train_weekly.idx       # 168h+24h 슬라이딩 윈도우 인덱스
    └── val_weekly.idx
```

### BB 프로토콜 준수

| 항목 | BB 프로토콜 | 우리 구현 |
|------|-----------|----------|
| 시계열 단위 | hourly total electricity (kWh) | ✅ 동일 |
| 정규화 | Box-Cox (global fitted) | ✅ 동일 |
| context | 168h (1주일) | ✅ 동일 |
| horizon | 24h (1일) | ✅ 동일 |
| loss | Gaussian NLL | ✅ 동일 |
| 평가 | median NRMSE, bootstrap 50K | ✅ 동일 |

## 9. 기대 효과

| 개선 요소 | 기대 NRMSE 개선 | 근거 |
|----------|:---:|------|
| 스케줄 다양성 (320→1,550+) | -2~3%p | baseload 분포 정상화 |
| 연식별 외피 (1→5 vintage) | -1~2%p | 에너지 패턴 다양성 |
| Baseload 보정 | -1~2%p | 실측 패턴에 근접 |
| **합계 기대** | **-4~7%p** | **21.31% → 14~17%** |

BB SOTA 13.31%와의 잔여 갭(1~4%p)은:
- 모델 아키텍처 개선 (PatchTST-L → BB Transformer-L)
- 학습 데이터 규모 (10K vs 900K)
로 추가 축소 가능.

## 10. 리스크 & 대안
